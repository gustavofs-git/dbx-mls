# Phase 02.1: Workspace Recreation — Eliminate NAT Gateway Cost — Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Delete the current Azure Databricks workspace and its managed resource group entirely (NAT gateway cannot be removed due to deny assignments), then recreate the workspace with public IPs on worker nodes — eliminating the NAT gateway and its R$35/month idle cost. Redeploy the full stack (SP admin, UC schemas, bundle deploy). Produce an article documenting the cost trap and fix.

This phase does NOT preserve existing data — Bronze tables are empty (pipeline failed before any ingestion). Fresh start is explicitly accepted.

</domain>

<decisions>
## Implementation Decisions

### Deletion method
- **D-01:** Use Azure CLI to delete — `az group delete` on the managed resource group, then `az databricks workspace delete` on the workspace resource group (or delete both resource groups via CLI)
- **D-02:** Delete order: managed RG first (contains NAT gateway, VNet, storage), then workspace RG (contains the workspace resource itself)

### Workspace recreation — cost-first config
- **D-03:** Recreate via Azure CLI (`az databricks workspace create`) — not the portal
- **D-04:** Explicitly set `--enable-no-public-ip false` (or omit `--no-public-ip`) so worker nodes receive public IPs — this prevents Azure from provisioning a NAT gateway
- **D-05:** Use the same region (`eastus`) and same subscription — no region change needed
- **D-06:** Use the lowest-cost SKU that supports Unity Catalog — `standard` tier is sufficient; avoid `premium` unless UC requires it (check at recreation time)
- **D-07:** No custom VNet injection — use Databricks-managed VNet, but with public IPs

### SP & OIDC continuity
- **D-08:** Reuse the existing App Registration (Service Principal) — it lives in Entra ID and survives workspace deletion entirely
- **D-09:** After workspace is created, re-add the SP as workspace admin via Databricks UI or CLI (`databricks permissions` or workspace settings)
- **D-10:** `AZURE_TENANT_ID` and `DATABRICKS_CLIENT_ID` GitHub repository variables do NOT change — they reference the Entra app, not the workspace
- **D-11:** `DATABRICKS_HOST` GitHub repository variable MUST be updated to the new workspace URL — this is the only GitHub secret that changes

### Bundle redeploy sequence (Claude's discretion)
Follow Phase 1 pattern exactly:
- **D-12:** Step 1 — Add SP as workspace admin (Databricks workspace settings)
- **D-13:** Step 2 — Create UC catalog and schemas owned by SP (`databricks bundle deploy` handles this via `resources/schemas.yml` if SP is admin)
- **D-14:** Step 3 — Update `DATABRICKS_HOST` in GitHub repository variables
- **D-15:** Step 4 — Push to `main` to trigger `cd-dev.yml` → `databricks bundle deploy --target dev`
- **D-16:** Riot API key secret scope must be recreated in new workspace (`databricks secrets create-scope` + `databricks secrets put`)

### Document and process updates (Claude's discretion)
- **D-17:** Update `docs/setup.md` (or equivalent) with the workspace recreation procedure and the `--enable-no-public-ip false` flag rationale
- **D-18:** Update STATE.md decisions log with the cost trap finding
- **D-19:** Update PROJECT.md Key Decisions table with the NAT gateway incident and mitigation

### Article
- **D-20:** Full LinkedIn article + short post required (`docs/posts/phase-2.1-article.md`, `docs/posts/phase-2.1-post.md`)
- **D-21:** Article covers: NAT gateway cost trap (R$35/month idle cost), why Azure deny assignments block manual removal, the solution (`--enable-no-public-ip false`), and the MCP-assisted debugging workflow used to diagnose the `__file__` error
- **D-22:** Tone: practical war story — "here's what happened and how I fixed it" — targets DE practitioners on Azure

## Summary

Phase 02.1 is a workspace rebuild operation driven by a cost issue: the current workspace was provisioned with `enableNoPublicIp=true` (the default as of ARM template API version 2024-05-01), causing Azure to automatically provision a NAT gateway in the managed resource group. The NAT gateway costs ~R$35/month even when idle. Because Azure applies deny assignments to managed resource groups, the NAT gateway cannot be manually removed — the workspace itself must be deleted and recreated with `--enable-no-public-ip false`.

**Critical discovery: D-06 is incorrect and must be overridden.** As of April 1, 2026, Azure no longer allows creation of Standard-tier workspaces. All new workspaces must use `--sku premium`. The planner must use `--sku premium` regardless of the CONTEXT.md statement about `standard` being sufficient. This is not a preference change — Standard workspace creation fails outright after April 1, 2026. The good news: existing workspaces are being auto-upgraded to Premium on October 1, 2026 anyway, so the phase should embrace Premium now.

The deletion sequence is straightforward using `az databricks workspace delete --force-deletion`: this single command deletes the workspace AND force-deletes the managed resource group (bypassing deny assignments and removing the workspace catalog storage). No managed-RG-first step is required. The deny assignments are automatically removed by the Databricks service when the workspace is deleted via the workspace delete API.

**Primary recommendation:** Delete with `az databricks workspace delete --force-deletion -y`, recreate with `az databricks workspace create --sku premium --enable-no-public-ip false --location eastus`, re-add SP as workspace admin via the Databricks Account Console, redeploy bundle, recreate secrets, update `DATABRICKS_HOST`.

**Deletion (single command, force-deletes managed RG):**
```bash
az databricks workspace delete \
  --name <WORKSPACE_NAME> \
  --resource-group <WORKSPACE_RG_NAME> \
  --force-deletion \
  --yes
```

Key flag notes:
- `--force-deletion`: "Optional parameter to retain default unity catalog data. By default the data will be retained if UC is enabled on the workspace." Using `--force-deletion` removes the storage/managed RG entirely. Since Bronze tables are empty (explicit fresh-start accepted per CONTEXT.md), use this flag.
- `--yes` / `-y`: Skip confirmation prompt
- No `--no-wait` needed unless you want async — wait for completion before creating new workspace

**Recreation (verified flag set):**
```bash
az databricks workspace create \
  --name <WORKSPACE_NAME> \
  --resource-group <WORKSPACE_RG_NAME> \
  --location eastus \
  --sku premium \
  --enable-no-public-ip false \
  --managed-resource-group <MANAGED_RG_NAME>
```

Key flag notes:
- `--sku premium`: Required (Standard tier creation blocked as of April 1, 2026)
- `--enable-no-public-ip false`: Accepted values are `{0, 1, f, false, n, no, t, true, y, yes}`. Setting `false` means worker nodes get public IPs — no NAT gateway is provisioned.
- `--managed-resource-group`: Optional but recommended so the managed RG gets a predictable name (e.g., `databricks-rg-<workspacename>-<location>`). If omitted, Azure generates a random name.
- `--compute-mode Hybrid`: Default; omit for standard job clusters. Do NOT use `Serverless` (not supported with custom configurations).
- No `--vnet`, `--private-subnet`, `--public-subnet` needed (D-07: use Databricks-managed VNet)

**Verify workspace creation:**
```bash
az databricks workspace show \
  --name <WORKSPACE_NAME> \
  --resource-group <WORKSPACE_RG_NAME> \
  --query "properties.workspaceUrl" \
  --output tsv
```

## Deletion Sequence — Corrected from D-01/D-02

**D-01 and D-02 describe a two-step delete (managed RG first, then workspace RG). This is unnecessary and wrong.**

Official Azure documentation (March 2026) states:

> "To force delete a workspace and its managed resource group using the Azure CLI, use the `az databricks workspace delete` command with the `--force-deletion` flag"

The `--force-deletion` flag in a single `az databricks workspace delete` call:
1. Deletes the Databricks workspace resource
2. Removes the deny assignment on the managed resource group
3. Deletes the managed resource group and all its contents (NAT gateway, VNet, storage account, access connector)

**Attempting to delete the managed RG first would fail** because the deny assignment blocks it. The workspace must be deleted first; the managed RG is cleaned up as part of that operation.

**Correct sequence:**
1. `az databricks workspace delete --force-deletion --name ... --resource-group ... --yes`
2. Wait for completion (the managed RG and NAT gateway are gone)
3. Proceed with `az databricks workspace create`

Source: [Delete a workspace — Azure Databricks Microsoft Learn](https://learn.microsoft.com/en-us/azure/databricks/admin/workspace/delete-workspace) — HIGH confidence (official docs, updated 2026-03-11)

## SP Admin Assignment — CLI vs UI

**Official method (UI — workspace admin settings):**

Source: [Manage service principals — Azure Databricks](https://learn.microsoft.com/en-us/azure/databricks/admin/users-groups/manage-service-principals) — HIGH confidence

There is no single Databricks CLI command to add a service principal as workspace admin directly. The official path uses either:

1. **Databricks Account Console UI** (most reliable):
   - Log into accounts.azuredatabricks.net
   - Workspaces → select workspace → Permissions tab → Add permissions → select SP → set Admin
   - **Then** in workspace Settings → Identity and access → Groups → admins group → Add members → add SP

2. **Workspace API** (automatable via curl/CLI):
   ```bash
   # First add SP to workspace (account-level assignment)
   # Uses: {account-domain}/api/2.0/accounts/{account_id}/workspaces/{workspace_id}/permissionassignments
   # Then add to admins group at workspace level
   ```
   The Databricks CLI does not have a high-level subcommand for "make SP workspace admin" — it requires direct API calls or UI.

3. **Two-step process always required:**
   - Step A: Add SP to workspace (account console or workspace settings)
   - Step B: Add SP to `admins` system group (workspace Settings → Identity and access → Groups → admins → Add members)

**D-09 says "via Databricks UI or CLI"** — the UI is the safest and clearest path for a human-gated step. The plan should specify UI steps explicitly since CLI API calls for this require account-level auth and SP numeric ID.

### DATABRICKS_HOST URL Will Change

Every new workspace gets a unique URL: `https://adb-<new-workspace-id>.<new-random>.azuredatabricks.net`

The workspace ID and random number are new on each workspace creation. **`DATABRICKS_HOST` must be updated** in GitHub repository variables after workspace creation.

- `AZURE_TENANT_ID` — unchanged (Entra ID tenant)
- `DATABRICKS_CLIENT_ID` — unchanged (App Registration UUID)
- `DATABRICKS_HOST` — **must be updated** (new workspace URL)

### OIDC Federation Policy Survives

The OIDC federation policies registered on the SP (dev branch-scoped + prod environment-scoped) are Databricks account-level entities, not workspace-level. They reference the Entra App Registration by SP numeric ID. They survive workspace deletion entirely.

No federation policy changes are required.

## Common Pitfalls

### Pitfall 1: Attempting to Delete Managed RG Before Workspace
**What goes wrong:** `az group delete` on the managed resource group returns permission denied or times out.
**Why it happens:** Azure applies system deny assignments to managed resource groups. Only the Databricks service (triggered by workspace deletion) can remove them.
**How to avoid:** Always delete via `az databricks workspace delete --force-deletion`. Never target the managed RG directly.
**Warning signs:** Error message containing "AuthorizationFailed" or "DenyAssignment" when attempting `az group delete` on a `databricks-rg-*` resource group.

### Pitfall 2: Using `--sku standard` on New Workspace Creation
**What goes wrong:** Command fails with a subscription-level policy error.
**Why it happens:** As of April 1, 2026, Standard-tier workspace creation is blocked in all Azure subscriptions.
**How to avoid:** Always use `--sku premium` for new workspace creation.
**Warning signs:** Plan or documentation still references D-06 ("standard is sufficient") — this must be explicitly overridden in the plan.

### Pitfall 3: Not Setting `--enable-no-public-ip false` (The Core Problem)
**What goes wrong:** New workspace is created with the default `enableNoPublicIp=true`, NAT gateway is provisioned again in managed RG, cost trap repeats.
**Why it happens:** `enableNoPublicIp=true` is the ARM template default since API version 2024-05-01. The CLI respects this default if the flag is omitted.
**How to avoid:** Explicitly pass `--enable-no-public-ip false` to the create command.
**Warning signs:** After workspace creation, check Azure Portal → managed RG for a NAT gateway resource. If present, the flag was not set correctly.

### Pitfall 4: Forgetting `--force-deletion` on Delete
**What goes wrong:** Workspace deletes but managed RG is retained (DBFS storage account and access connector remain). The managed RG still costs money. A new workspace creation may conflict if the same managed RG name is reused.
**Why it happens:** Per official docs: "When you delete an Azure Databricks workspace, most compute resources are cleaned up. However, the DBFS storage account and access connector in the managed resource group are retained unless you force delete them."
**How to avoid:** Always use `--force-deletion` when the intent is complete cleanup. This is safe here because Bronze tables are empty (explicit fresh start per CONTEXT.md).

### Pitfall 5: Updating `DATABRICKS_HOST` Before SP is Workspace Admin
**What goes wrong:** cd-dev.yml triggers and `databricks bundle deploy` fails with 403 — SP has no workspace access yet.
**Why it happens:** The GitHub Actions workflow uses OIDC to authenticate as the SP. If the SP is not yet added to the workspace, all API calls fail.
**How to avoid:** Follow the sequence strictly: SP added as workspace admin (Steps 5-6) before updating DATABRICKS_HOST (Step 10) before pushing to main (Step 11).

### Pitfall 6: Missing SP READ ACL on `lol-pipeline` Scope
**What goes wrong:** Bundle deploy succeeds but ingestion job fails at runtime with `PERMISSION_DENIED` when `dbutils.secrets.get('lol-pipeline', 'riot-api-key')` is called.
**Why it happens:** Default scope creator is the human admin. The SP running the job has no READ ACL until explicitly granted.
**How to avoid:** Run `databricks secrets put-acl lol-pipeline <sp-uuid> READ` after creating the scope.

### Pitfall 7: DATABRICKS_HOST Workspace ID Confusion
**What goes wrong:** Developer copies an old URL or misreads the new workspace ID.
**Why it happens:** The new URL is `adb-<new-workspace-id>.<new-random>.azuredatabricks.net`. Both the numeric workspace ID and the random number are unique per workspace — the old URL will 404.
**How to avoid:** Use `az databricks workspace show --query "properties.workspaceUrl" --output tsv` to get the exact URL after creation.

## Runtime State Inventory

This is a delete-and-recreate operation. The inventory of what survives vs. what is destroyed is:

| Category | Items | Action Required |
|----------|-------|-----------------|
| Stored data (Bronze tables) | Empty — pipeline failed before any ingestion (per CONTEXT.md: "fresh start is explicitly accepted") | No data migration needed |
| Azure workspace resource | Deleted by `az databricks workspace delete --force-deletion` | Recreate with new command |
| Managed resource group (NAT gateway, VNet, storage) | Deleted by `--force-deletion` flag | No separate action; handled by workspace delete |
| Databricks Secret Scope `lol-pipeline` | Deleted with workspace (workspace-level resource) | Recreate: `create-scope` + `put-secret` + `put-acl` |
| OIDC federation policies | Survive — they live at Databricks account level, not workspace level | No action required |
| SP / App Registration | Survives in Entra ID — unaffected by workspace deletion | Re-add to new workspace as admin |
| `DATABRICKS_HOST` GitHub variable | Stale after deletion — new workspace gets new URL | Update in GitHub repo variables |
| `AZURE_TENANT_ID` GitHub variable | Unchanged | No action required |
| `DATABRICKS_CLIENT_ID` GitHub variable | Unchanged | No action required |
| Git repository / code | Unchanged — `__file__` fix already committed | No action required |
| `databricks.yml` bundle config | Unchanged — no `workspace.host` set (uses env var) | No action required |

**The canonical question answer:** After all files are updated, the only runtime state with the "old workspace" is:
1. `DATABRICKS_HOST` GitHub variable → must be updated to new URL
2. `lol-pipeline` secret scope → must be recreated from scratch
3. SP workspace membership → must be re-added to new workspace

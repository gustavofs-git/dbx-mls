**H1 Title:** Something direct about the cost trap and fix — e.g., "The Azure Databricks NAT Gateway I Didn't Order: A One-Flag Fix and What It Took to Get There"

**Section 1 — The Unexpected Bill**
Opening hook: a NAT gateway appeared in my managed resource group on workspace creation. I didn't provision it. It costs approximately R$35/month (Azure East US, Q1 2026 pricing) even when completely idle — no traffic, no clusters running, no data. Just existing. Explain that `enableNoPublicIp=true` became the ARM API default as of API version 2024-05-01. This setting routes all outbound worker node traffic through a NAT gateway for "secure cluster connectivity." Azure automatically provisions the NAT gateway in the managed resource group. There is no prompt. No warning in the portal. No cost estimate shown during workspace creation.

**Section 2 — Why You Can't Just Delete It**
Azure applies deny assignments to Databricks managed resource groups. Even as subscription owner, `az group delete` on the `databricks-rg-*` resource group fails with `AuthorizationFailed`. This is intentional — the Databricks service manages that resource group's lifecycle, and protecting it from accidental deletion is the design intent. But the side effect is that you cannot surgically remove the NAT gateway without deleting the workspace itself. Mention the error message pattern developers will hit: `DenyAssignment` or `AuthorizationFailed`. No workarounds exist — the deny assignment can only be removed by the Databricks service itself, which happens as part of workspace deletion.

**Section 3 — The MCP Debugging Session**
Brief mention (2-3 paragraphs): During the same debugging session, the `__file__` AttributeError surfaced — a NameError that occurs when Databricks notebook execution context tries to resolve `__file__` (which doesn't exist in the cell execution runtime). The Databricks MCP (Azure Databricks MCP ai-dev-kit via Claude Code) was used to inspect the workspace, review the job run error output, and trace the error to the sys.path bootstrap pattern on line 1 of every ingestion notebook. The fix was `os.getcwd()` instead of `os.path.dirname(os.path.abspath(__file__))` — two lines changed across six notebooks. This demonstrates the AI-assisted DE workflow pattern: MCP provides workspace access directly in the IDE, no context-switching to the Databricks UI.

**Section 4 — The Fix: One Flag**
The fix for the NAT gateway cost trap is a single CLI flag on workspace creation: `--enable-no-public-ip false`. Show the full delete and recreate commands with all flags explained:

Delete (single command):
```bash
az databricks workspace delete \
  --name <WORKSPACE_NAME> \
  --resource-group <WORKSPACE_RG_NAME> \
  --force-deletion \
  --yes
```

Recreate (with the flag that prevents NAT gateway):
```bash
az databricks workspace create \
  --name <WORKSPACE_NAME> \
  --resource-group <WORKSPACE_RG_NAME> \
  --location eastus \
  --sku premium \
  --enable-no-public-ip false \
  --managed-resource-group <MANAGED_RG_NAME>
```

Explain each flag: `--force-deletion` removes the deny assignment and cleans up the managed RG entirely; `--sku premium` is required as of April 1, 2026 (Standard SKU creation is blocked — existing workspaces auto-upgrade October 2026); `--enable-no-public-ip false` is the explicit override that prevents the NAT gateway.

Verify no NAT gateway was created:
```bash
az resource list \
  --resource-group <MANAGED_RG_NAME> \
  --query "[?type=='Microsoft.Network/natGateways'].name" \
  --output tsv
# Expected: empty output
```

**Section 5 — The Rebuild Checklist**
After workspace deletion, runtime state that must be restored (SP App Registration, OIDC federation policies, and GitHub variables `AZURE_TENANT_ID`/`DATABRICKS_CLIENT_ID` survive — only `DATABRICKS_HOST` changes):

1. Re-add SP as workspace admin (Databricks Account Console → Workspaces → Permissions)
2. Add SP to workspace `admins` group (workspace Settings → Identity and access → Groups)
3. Recreate secret scope:
   ```bash
   databricks secrets create-scope lol-pipeline
   databricks secrets put-secret lol-pipeline riot-api-key --string-value "<key>"
   databricks secrets put-acl lol-pipeline "<sp-uuid>" READ
   ```
4. Update `DATABRICKS_HOST` GitHub variable to new workspace URL (`az databricks workspace show --query "properties.workspaceUrl"`)
5. Push to `main` → `cd-dev.yml` triggers → `databricks bundle deploy --target dev --auto-approve`

**Section 6 — Takeaway**
Always inspect the managed resource group immediately after workspace creation. Check for `Microsoft.Network/natGateways` — it should not be there for a dev workspace. The fix is cheap (a single flag) but recovery requires deleting the workspace, which on this project meant a debugging session to confirm no data worth preserving. Build the habit of verifying managed RG contents after provisioning, before doing any real work.
